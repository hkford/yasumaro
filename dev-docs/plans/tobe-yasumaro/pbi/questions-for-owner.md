## 優先度高（実装をブロックする質問）

### Q1. fetchWithRetry のタイムアウト時リトライ回数（PBI-107）

**背景**: 現在 OpenAI・Gemini ともに `maxRetryCount: 3`。タイムアウト時も3回リトライしている。

**BDD 要件**: タイムアウト時は1回のみリトライ、429（レートリミット）時は0回。

**質問**: 以下の変更方針でよいか？
- `AbortError`（タイムアウト）: 最大1回リトライ（合計2回）
- HTTP 429: リトライなし（即座にエラー返却）
- HTTP 5xx: 変更なし（最大3回リトライ）

**実装上の懸念**: `src/utils/fetch.ts` の `fetchWithRetry` 関数を変更すると全AIプロバイダーに影響する。テスト接続（現在 `maxRetryCount: 1`）は個別設定なので影響なし。

解答：OK

### Q2. プライバシー同意「拒否」時の「制限モード」の定義（PBI-102）

**背景**: BDD シナリオに「拒否時は拡張機能が制限モードで起動する」と記載されているが、「制限モード」の具体的な機能範囲が定義されていない。

**現状の実装**: 3回拒否すると以降モーダルが表示されない（永続的にdismiss）。機能制限は行っていない。

**質問**: 「制限モード」で何を無効化するのか？
- [ ] AIサマリー生成を無効化する
- [ ] Obsidian への自動保存を無効化する
- [ ] SQLiteへのローカル保存も無効化する（記録停止）
- [ ] 現状のまま「モーダルが出ないだけ」で十分

**判断理由が必要な理由**: 機能を無効化する場合、service-worker.ts のメッセージハンドラに分岐を追加する工数が発生する。「モーダルが出ないだけ」なら現状で完了扱いにできる。

解答：
AIサマリー生成を無効化する かつ Obsidian への自動保存を無効化する かつ SQLiteへのローカル保存も無効化する

## 優先度中（実装の方向性に影響する質問）

### Q3. ブランド改名通知バナーの要否（PBI-103）

**背景**: BDD シナリオに「旧版（Obsidian Weave）をインストールしているユーザーへの通知バナー表示」が含まれているが、`src/` 内に該当実装が見当たらない。

**質問**: この通知バナーは今回の PBI で実装する必要があるか？

**背景情報**:
- 旧パッケージ名は `obsidian-weave`（Chrome Web Store で別の拡張機能として公開されていた場合）
- Chrome Extension は異なる Extension ID を持つため、「旧版がインストールされているか」の検出は難しい
- 実装するなら `chrome.runtime.onInstalled` で初回インストール時のみバナーを1度表示する方法が現実的

**コストの目安**: 実装するなら 2〜3 時間程度（バナーUI + 表示済みフラグ管理）


解答： CHrome Web Store では obsidian-weave として公開していない。そのため考慮しなくてよい。

### Q4. OPFS復旧時の自動マイグレーション要否（PBI-106）

**背景**: BDD シナリオに「OPFS復旧時にchrome.storage.localからSQLiteへ自動マイグレーション」とあるが、未実装。

**質問**: この機能は今回のPBIで必要か？

**背景情報**:
- モバイルChrome で OPFS が使えないブラウザバージョンが「後でアップデートされてOPFSが使えるようになる」ケースを想定
- 既存の `migrationService.ts` は Chrome.storage.local → SQLite のマイグレーションのみ対応（OPFS フォールバック → SQLite は未対応）
- 実装コストは高め（既存の migrationService を拡張する or 専用の復旧フローを作る）

**コストの目安**: 実装するなら 5〜8 時間程度（マイグレーション + エラーリカバリ）


解答： 必要。
