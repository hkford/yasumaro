# これは何？

- obsidian-weaveとして開発をしてきた。
  - Obsidian Local REST APIを経由したローカルVaultへの保存、Service Worker（SessionStore）による状態永続化を特徴にしてきた。
  - その結果、Obsidian Local REST APIにたよっている。
- ゴール
  - Obsidianがなくても、Chrome拡張として、ユーザがAI利用のためのAPI登録だけで利用できること
  - Obsidianなしでも、sqliteを使用することで使える
  - Obsidianが動作していれば、Obsidian RESTをつかって、（これまで通り）Obsidianへの出力機能を有する。
  - 最終的に、chrome web storeで利用できるように発表する。
- Obsidianがなくても動作するようにはしたいが、現在のobsidian-weaveが備える機能は全て保持する。

## プロジェクト名は、'yasumaro' とする。

- なぜ名前をかえるのか？それは、obsidan-weave という名前で公開するのが困難であると考えるから。
  - MinecraftのMod（Obsidian Weave）: マイクラの特定のMod環境で使われる共有コード用のライブラリ。
  - なんてのもあるし、Obsidian公式コミュニティプラグインの『Weave』もあるので。
- したがって、githubのレポジトリ名もrenameする予定である。

### 太安万侶（おおのやすまろ）は古事記の編纂者

―― 記憶の言葉を、永遠の「文字」に刻んだ冷徹な記録者

稗田阿礼が「記憶する者」なら、太安万侶はそれを「記録（システム）に落とし込んだ者」です。

特徴・エピソード: 元明天皇の命を受け、稗田阿礼が暗誦した膨大な記憶を文字に起こして『古事記』を完成させました。当時は日本語を書き表す正確な文字がなかったため、漢字を独自のルールで組み合わせて日本語の響きやニュアンスを表現するという、天才的な編集・プログラミング能力を発揮しました。

イメージ: 書記、編集者、クリエイター、無形の記憶を有形にする者。

ってことで  yasumaro と命名する。

既存のツール（Minecraft Modや公式プラグイン）との名前の重複を完全に回避できるだけでなく、「流動的な記憶（ブラウジング体験）を、普遍的な記録（構造化データ）に変換する」

太安万侶のエピソードをこのシステムに当てはめると、開発のロードマップが非常にクリアになります。

* **稗田阿礼（＝ブラウザ/ユーザーの体験）**: 日々大量のWebサイトを読み、タブを閉じ、消費していく「流動的で、放っておけば消えてしまう記憶」。
* **太安万侶（＝yasumaroシステム）**: それをキャッチし、独自のルール（AI要約・メタデータ付与）で、後から検索・活用できる「不変の記録」として刻む。

特に「当時はなかった表記法（独自のルール）を編み出して、日本語のニュアンスを文字に落とし込んだ」**という点が秀逸です。Web上の混沌としたHTMLや、AIの自由な出力を、**「SQLiteのスキーマ」という現代の文字（構造）に落とし込む行為そのもの

## 「Obsidianなし（SQLite永続化）」にする圧倒的メリット

ObsidianのLocal REST APIに依存せず、裏側でSQLiteを走らせる（または拡張機能内・ローカルサーバーでSQLiteに保存する）アプローチは、アーキテクチャとしても非常に理にかなっています。

## `obsidian-weave` の主要コンポーネント（これは基本的に保持する）

客観的に見て、このツールは以下の4つのコア要素で構成されています。

* **データ収集・制御**: ブラウザ閲覧履歴の自動/手動保存、ドメインフィルタ、サイトごとの許可設定。
* **テキスト処理（クレンジング）**: リンク密度フィルタ、MMR重複除去など、LLMに送るトークンを削減するロジック。
* **AI・メタデータ連携**: 各種LLM API（Groq, Gemini, OpenAI, Ollama等）による要約、カテゴリタグの自動生成、Prompt Injection対策。
* **ストレージ・同期基盤**: Obsidian Local REST APIを経由したローカルVaultへの保存、Service Worker（SessionStore）による状態永続化。→ Obsidian Local REST APIにたよっている。

### ツインエンジン（SQLite + Obsidian）を積んだ `yasumaro` のデータフロー

```
[ ブラウザで50％以上、5秒以上、特定のページを見るなどがイベントトリガー ]
           ↓ (イベント発生)
[ AIが内容を要約・タグ付け ]
           ↓
 ┌─────────┴─────────┐
 ▼ (メイン：常に実行)     ▼ (サブ：条件付き)
【Wasm SQLite (OPFS)】   【Obsidian Local REST API】
 ・すべての履歴を保存      ・Obsidianが起動＆設定されていれば
 ・高速全文検索用          ・デイリーノート等に自動で追記
 ・ダッシュボードで閲覧
```

1. **一次記録（万能の書庫）:** イベントトリガーがあると、AIの要約データは**必ず**拡張機能内のSQLite（OPFS）に一瞬で保存されます。Obsidianが閉じていようが、PCがオフライン（ローカルAI使用時など）だろうが、あなたのブラウジング史は1つも漏らさずローカルに蓄積されます。
2. **自動流し込み（Obsidianへの架け橋）:**

   バックグラウンド（Service Worker）がObsidianのLocal REST APIへの接続をトライします。Obsidianが起動していれば、これまで通りデイリーノートなどにデータを綺麗に流し込みます。もしObsidianが閉じていれば、エラーで落ちるのではなく「今回はSQLiteのみに保存」としてスマートに処理を終えます。

### ダッシュボード画面（B）が果たす「編纂」の役割

全画面のダッシュボード（B）があることで、データの価値が何倍にも跳ね上がります。

* **「検索・振り返り」の場:**

  「3日前に見た、あの記事なんだっけ」をSQLiteから爆速で全文検索。カレンダーやタイムライン形式で、自分の思考の軌跡を美しくビジュアル化して振り返ることができます。
* **「手動での再編纂（手動エクスポート）」:**

  もし自動流し込みのタイミングでObsidianが閉じていたログや、「これはObsidianではなくNotionに持って行きたい」「この記事だけ別の専用ノートとして切り出したい」という場合、ダッシュボード上でログを選択し、ワンクリックで最適なMarkdown形式に整形してクリップボードにコピー（またはファイル出力）できます。

### 開発・設計の視点でのメリット

この設計なら、現在の `obsidian-weave` のソースコード（タブ閉鎖の検知、AI要約のプロンプト、Obsidian APIを叩くロジックなど）の主要部分を活かしながら、「保存先としてSQLiteを追加する」という形で、段階的に `yasumaro` へとリファクタリング（移行）していくことができます。

## forever records

現状では1万件だか1週間だかで、obsidian-weaveからはURLの履歴を消している。これを消さないようにする(消す機能はダッシュボード内につける)

自動消去を廃止して「すべて残す（蓄積する）」方向、これこそSQLite（Wasm + OPFS）への移行が最も活きる部分ですね！

これまでの「1万件」や「1週間」という制限は、おそらく `chrome.storage.local` の容量制限（デフォルトで約5MB〜10MB）や、データが増えたときのパフォーマス低下（配列の全走査など）を避けるための防衛策だったのだと思います。

しかし、**SQLite + OPFS の組み合わせであれば、件数が10万件、100万件になってもインデックス検索によって一瞬でデータが引けるため、自動で消す必要は全くなくなります。** ユーザーにとっては「自分の全ブラウジング史が一生モノの資産として溜まっていく」という、圧倒的な価値に変わります。

この仕様変更に伴い、技術的に対応すべきポイントと、ダッシュボードに搭載する「消す機能」の設計アイデアを整理しました。

### 1\. 「消さない」を実現するための技術的対応

拡張機能の中でデータを無限に溜められるようにするため、以下の2点が必要になります。

* `manifest.json` への権限追加:

  Chrome拡張機能で容量制限をなくすため、`permissions` に `"unlimitedStorage"` を追加します。これがないと、データが溜まったときにブラウザ側で書き込みエラーが発生します。

  JSON

  ```
  "permissions": [
    "unlimitedStorage",
    "storage",
    "offscreen" // Wasm SQLiteを動かす場合に必要
  ]
  
  ```
* **SQLiteのインデックス設計:**

  URLやタイトル、保存日時（`created_at`）にしっかりインデックス（INDEX）を貼っておきます。これにより、データが何万件に増えても、ダッシュボードの読み込みや検索が重くなるのを完全に防げます。

### 2\. ダッシュボード内に搭載する「消す機能（データ管理）」のアイデア

自動で消さなくなる分、ユーザーが「自分でコントロールできる安心感」をダッシュボード内で提供するのが良さそうです。以下のような削除・管理オプションが考えられます。

* **個別削除 / 選択削除:**

  タイムラインに並んだログから、「これは残さなくていいな」というものを1件ずつ、あるいはチェックボックスで複数選択してゴミ箱ボタンで消せるようにします。
  * *内部処理:* `DELETE FROM logs WHERE id = ?;`
* **条件を指定した一括削除（クリーンアップ）:**

  「1ヶ月以上前のログを一括削除」「特定のドメイン（例: `youtube.com` や社内ツールのURLなど）のログを一括削除」といった、フィルターをかけた削除機能です。
* **「お気に入り（ピン留め）」機能の併用:**

  間違えて大事な要約を消してしまわないよう、ログに「星（お気に入り）」をつけられるようにします。一括削除を実行しても「星付きのログは保護されて残る」という仕様にすると、ユーザーが安心してクリーンアップを行えます。

### 3\. 次のステップとしてのデータ移行（マイグレーション）

もし、すでに現在の `obsidian-weave` を使っているユーザー（あるいはあなた自身）がいる場合、独立した `yasumaro` にアップデートした最初の起動時に、「既存の `chrome.storage.local` にある過去の履歴（1万件分など）を、自動で新しいSQLiteにインサートして引き継ぐ」というスクリプトを1回だけ走らせると、これまでのデータを無駄にせずスムーズに移行できます。

「消さずに、自分の人生のログとしてすべて蓄積していく」という仕様になったことで、`yasumaro`（太安万侶）という『普遍の記録を刻む者』のキャラクター性がさらに強くなりましたね。

### 2\. 公開までの4つのステップ

#### ステップ1：Chromeデベロッパーダッシュボードに登録

1. [Chrome Developer Dashboard](<https://chrome.google.com/webstore/devconsole/?authuser=3>) にアクセスします。
2. Googleアカウントでログインし、デベロッパー規約に同意します。
3. 登録料（5ドル）をクレジットカードで支払います（※1回支払えば、以降は無制限に拡張機能を公開できます）。

#### ステップ2：パッケージ（ZIPファイル）のアップロード

1. ダッシュボードの「新しいアイテムを追加」をクリックします。
2. `yasumaro` のソースコード（`manifest.json` がルートにある状態）をZIPに固めてアップロードします。
3. アップロードが成功すると、マニフェストのエラーがないか自動チェックが行われます。

#### ステップ3：ストア掲載情報の入力

* **表示名・説明文:** ユーザーに向けて `yasumaro` の魅力を記載します。
* **カテゴリ:** 「生産性（Productivity）」や「開発者ツール（Developer Tools）」が最適です。
* **グラフィック資産:** 準備したアイコンやスクリーンショットをアップロードします。

#### ステップ4：プライバシーとパーミッションの申告（★重要）

ユーザーのデータをどのように扱うかをGoogleに申告する「プライバシー（Privacy practices）」タブの入力が必要です。

* **単一の目的（Single Purpose）:** 「ブラウジング履歴をローカルのSQLiteに保存し、AIで要約・管理するため」と明記します。
* **データの利用目的:** 「ユーザーのデータを外部のサーバーに送信して収集することはない（ローカル完結）」ことを強くアピールします。

### 3\. `yasumaro` 特有の審査対策（リジェクトを避けるコツ）

`yasumaro` は「タブの情報を取得する」「大容量のローカル保存を行う」「外部AIのAPIを叩く」という特性上、Googleの審査（Review）で厳しく見られる可能性があります。以下の2点に注意して申請してください。

#### ① 権限の「正当な理由（Justification）」を明確に書く

ダッシュボード内で、なぜそのパーミッションが必要なのかを説明する欄があります。

* `"unlimitedStorage"`: 「ユーザーが過去のブラウジング履歴やAI要約を制限なくローカルのSQLiteに蓄積できるようにするため」
* `"offscreen"`（Wasm SQLiteを使う場合）: 「Manifest V3のService Workerの制限を回避し、バックグラウンドで安全にSQLiteの書き込み処理を行うため」
* `"tabs"` または `"activeTab"`: 「ユーザーが閉じたタブのURLやタイトル、テキストを取得して要約を生成するため」

#### ② APIキーの扱い

ユーザー自身が自分のAPIキー（GeminiやOpenAIなど）を入力して使う仕組みであることを説明文などに補足しておくと、拡張機能自体が怪しい通信を行っていると誤解されるのを防げます。

### 4\. 審査から公開まで

すべての入力が終わったら、「審査に提出（Submit for Review）」をクリックします。

現在のChrome Web Storeの審査は、通常 **2日〜1週間程度** で完了します（パーミッションが多い拡張機能は少し時間がかかる傾向にあります）。審査が通れば、自動的に世界中のChromeユーザーがインストールできるようになります！
