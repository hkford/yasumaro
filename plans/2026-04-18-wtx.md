## 3. 気がついたこと

### 问题1: WXTは собственных API を强制する

```
Error: Default export not found, did you forget to call "export default defineBackground(...)"?
```

- WXTは `defineBackground()`, `defineContentScript()` などの专用API使用を要求
- 既存の `src/` コード（约5000行）はこれらのAPIを使用していない
- 移行には全エントリーポイントの 书き直しが必要

### 问题2: ファイルパスの複雑さ

```
Error: Failed to resolve ../popup/i18n.js from entrypoints/options/index.html
Error: Failed to resolve privacy.js from entrypoints/permissions/index.html
```

- 既存のHTMLは `../popup/i18n.js`, `../icons/icon48.png` などの相对パスを参照
- entrypoints/结构に移行すると全パスの修正が必要
- icons, CSS, i18nなど多个静态アセットの再配置问题

### 问题3: ディレクトリ结构の非兼容

WXT期待:
```
entrypoints/
├── popup/
│   ├── index.html
│   └── main.ts
├── background/
│   └── index.ts
└── content/
    ├── loader.ts
    └── extractor.ts
```

现在的结构:
```
src/
├── popup/
│   ├── popup.html
│   └── main.ts
├── background/
│   └── service-worker.ts
└── content/
    ├── loader.ts
    └── extractor.ts
```

### 问题4: サービスワーカーの import 方式

```typescript
// WXT期待する方式
export default defineBackground({
  main() {
    import('../../src/background/service-worker.js');
  },
});

// 现在的方式
// src/background/service-worker.ts が直接chrome.runtime.onMessageをlisten
```

## 6. 教訓

1. **WXTは「_FRAMEWORK」**: 单なる插件ではなく、思想的な差异が大きい
2. **导入问题**: `import { defineBackground } from 'wxt'` は动作しない → `'wxt/utils/define-background'` が必要
3. **移行费用対効果**: 既存の稳定构成を変えられるほどではありませんでした
4. **别のタイミングで**: 全新規開発時にWXT选择肢として考虑的
