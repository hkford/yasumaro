# PBI: ダッシュボードUI（カレンダー+タイムラインビュー・全文検索）

## ユーザーストーリー
**yasumaroユーザーとして**、全画面の新規タブでブラウジング履歴をカレンダーとリスト形式で振り返り、全文検索で過去の記事を素早く見つけたい、なぜなら自分の知識の軌跡を「資産」として活用したいから。

## ビジネス価値
- Chrome Web Storeでの差別化ポイント：ビジュアルなタイムラインで過去の閲覧を「体験」として振り返れる
- 全文検索により「あの記事どこだっけ」問題を解決し、ユーザーのリテンションを向上

## BDD受け入れシナリオ

```gherkin
Feature: ダッシュボードUI

Scenario: カレンダーで日付を選択してその日の記録を見る
  Given SQLiteに過去30日分のブラウジングログが存在する
  When ユーザーがダッシュボードを開き、カレンダーで3日前の日付をクリックする
  Then その日のブラウジングログが時系列リストで表示される
  And 各ログにはタイトル・ドメイン・AI要約・記録時刻が表示される
  And 記録がある日付はカレンダー上でハイライトされている

Scenario: 全文検索で過去のログを検索する
  Given SQLiteに1000件のブラウジングログが存在する
  When ユーザーが検索ボックスに "TypeScript 型ガード" と入力する
  Then FTS5検索結果が50ms以内に表示される
  And 検索キーワードがハイライトされた状態で結果が表示される
  And 「N件見つかりました」という件数が表示される

Scenario: ログを個別削除する
  Given ダッシュボードにブラウジングログが表示されている
  When ユーザーが特定のログのゴミ箱アイコンをクリックして削除を確認する
  Then そのログがSQLiteから論理削除される（is_deleted=1）
  And リストからそのログが即座に消える

Scenario: ログにスター（お気に入り）をつける
  Given ダッシュボードにブラウジングログが表示されている
  When ユーザーが特定のログのスターアイコンをクリックする
  Then そのログのis_starredが1に更新される
  And スターフィルターで絞り込むとそのログのみが表示される
```

## 受け入れ基準

## テスト戦略（t_wadaスタイル）

### E2Eテスト（Playwright）
- ダッシュボードを開き、カレンダーで日付クリック → リスト表示確認
- 検索ボックスにキーワード入力 → 結果表示確認
- ログのスター/削除操作

### 統合テスト
- `DashboardDataService.getLogsByDate(date)` がSQLiteから正しいデータを返す
- `DashboardDataService.search(keyword)` がFTS5経由で結果を返す
- `DashboardDataService.updateStar(id, starred)` がSQLiteを更新する

### 単体テスト
- カレンダーコンポーネント: 「記録あり日付」のハイライトロジック
- 検索デバウンス: 300ms以内の連続入力では1回だけ検索が発火する
- 論理削除: is_deleted=1のレコードがリストに表示されない

## 実装アプローチ
- **Outside-In**: Playwright E2Eテストからシナリオを定義し、統合テスト → UIコンポーネント実装の順
- **UI設計先行**: ワイヤーフレームをHTMLモックで先に作り、ユーザー確認後にロジックを接続する

## 技術的考慮事項
- 依存関係: Phase 1（SQLiteコア基盤）完了が前提
- 既存 `src/dashboard/dashboard.html` / `dashboard.ts` のパターンを踏襲
- カレンダーライブラリ: 軽量なものを選定（例: vanilla JS実装か `@fullcalendar` 系）
- CSP制約: inline script禁止のため、全イベントハンドラはTypeScriptから登録する
- `manifest.json` の `web_accessible_resources` にダッシュボード関連JSを追加すること

### 落とし穴
- `manifest.json` の `chrome_url_overrides` でnewtabを上書きする方法もあるが、ユーザー体験を壊す可能性があるため `chrome.tabs.create()` 方式を推奨
- FTS5の検索クエリはSQL Injectionに注意（プリペアドステートメントを必ず使用）
- 大量データ表示時は仮想スクロール（Virtual Scroll）の実装を検討すること
