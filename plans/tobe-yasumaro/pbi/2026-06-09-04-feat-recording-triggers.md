# PBI: 記録トリガー設定UI（ユーザーが複数条件から選択）

## ユーザーストーリー
**yasumaroユーザーとして**、いつ・どの条件でページが記録されるかを自分で選択・設定したい、なぜなら記録頻度と精度を自分のライフスタイルに合わせてカスタマイズしたいから。

## ビジネス価値
- 記録条件をカスタマイズできることで「意図しない記録」「記録漏れ」の不満を解消する
- パワーユーザーから初心者まで多様なニーズに対応し、リテンション向上につながる

## BDD受け入れシナリオ

```gherkin
Feature: 記録トリガー設定

Scenario: ユーザーが複数のトリガーを有効化する
  Given ダッシュボードの設定画面が開いている
  When ユーザーが「タブクローズ時」「読み対済50%+5秒以上」「手動保存」の3つを選択して保存する
  Then 選択した3つのトリガーが chrome.storage.local に保存される
  And 次のページ閲覧から、設定された条件でのみ記録が行われる

Scenario: タブクローズ時トリガーが動作する
  Given 「タブクローズ時」トリガーが有効になっている
  When ユーザーが https://example.com を閲覧中にタブを閉じる
  Then そのページのAI要約処理が開始され、SQLiteに記録される

Scenario: スクロール+滞在時間トリガーが動作する
  Given 「読み対済50%+5秒以上」トリガーが有効になっている
  When ユーザーがページの50%以上をスクロールし、5秒以上滞在した後にタブを離れる
  Then そのページが記録対象としてマークされ、SQLiteに保存される

Scenario: 定期スナップショットトリガーが動作する
  Given 「定期スナップショット（5分ごと）」トリガーが有効になっている
  When 5分が経過する
  Then 現在アクティブなタブのURLとタイトルが記録される

Scenario: 手動保存トリガーでポップアップから保存する
  Given 「手動保存」トリガーが有効になっている
  When ユーザーがポップアップを開いて「今すぐ保存」ボタンをクリックする
  Then 現在のアクティブタブのページが即座に記録される
  And 「保存しました」というフィードバックがポップアップに表示される

Scenario: トリガーが1つも選択されていない場合
  Given ユーザーが全てのトリガーを無効化して保存しようとする
  When 保存ボタンをクリックする
  Then 「少なくとも1つのトリガーを選択してください」というバリデーションエラーが表示される
  And 設定は保存されない
```

## 受け入れ基準
- [ ] 設定画面（ダッシュボード内）で以下のトリガーが複数選択可能:
  - タブクローズ時（デフォルト: ON）
  - 読み対済50%+5秒以上（デフォルト: OFF）
  - 手動保存（ポップアップから）（デフォルト: ON）
  - 定期スナップショット（デフォルト: OFF、間隔を分単位で設定可能）
- [ ] 少なくとも1つのトリガーが必須（バリデーション）
- [ ] 定期スナップショットは `chrome.alarms` で実装する（Service Worker再起動に対応）
- [ ] トリガー設定は `StorageKeys.RECORDING_TRIGGERS` に保存する
- [ ] 既存の「コンテンツスクリプトによる滞在時間計測」ロジックをトリガー判定に利用する
- [ ] i18n対応（日本語・英語）

## テスト戦略（t_wadaスタイル）

### E2Eテスト（Playwright）
- ダッシュボードの設定画面でトリガーを変更→保存→拡張機能の実際の動作確認

### 統合テスト
- `RecordingTriggerManager.shouldRecord(event)` が設定に基づいて正しくbooleanを返す
- `chrome.alarms` のセットアップ/クリーンアップが正しく動作する

### 単体テスト
- トリガー設定のバリデーション（0個選択時のエラー）
- スクロール率・滞在時間の閾値判定ロジック
- スナップショット間隔の境界値テスト（最小1分、最大60分）

## 実装者向け注記

### 現状コードの確認
```bash
grep -rn "shouldRecord\|scroll\|visitDuration\|engagement" src/background/recordingLogic.ts
grep -rn "chrome.alarms" src/background/
grep -rn "scroll\|duration\|engagement" src/content/
```

### 落とし穴
- `chrome.alarms` のコールバックはService Workerを起動させるが、wa-sqliteのOffscreen Documentも同時に初期化済みである必要がある。初期化を await してから保存すること
- 定期スナップショットで同じURLを短時間に重複保存しないよう、直近N分以内の重複チェックを行うこと
